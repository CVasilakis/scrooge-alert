# Scrooge Alert - Development Context

## Project Overview
This project is an automated Python application designed to monitor product prices across Skroutz domains and send push notifications when prices drop below user-defined target thresholds. 

- **Main Technologies:** Python 3, `apprise` (for versatile push notifications via Telegram, Discord, etc.), `tls_client` (for evasive web scraping by mimicking browser TLS fingerprints), and systemd (for automated background scheduling on Linux).
- **Architecture:** The application features a modular, plugin-based architecture residing in `src/core/`. The entrypoint is `main.py`, which delegates to specialized modules:
  - **Scrapers (`scrapers/`)**: Self-contained plugin packages grouped by target store. Each plugin (e.g. `scrapers/skroutz/`) contains its own client, model, storage, an optional `requirements.txt` (its private dependencies), and a `plugin.py` descriptor that acts as the single source of truth for domains, config filenames, class bindings, its private dependency list (`get_requirements_path()`), its *default* systemd schedule (`get_timer_directives()`), and its settings class (`get_settings_class()`). **Import-light contract (load-bearing):** `plugin.py` and the package `__init__` are imported for *every* plugin during discovery, so they must import only stdlib + the base contracts — never a transport/parsing library (`tls_client`, `selenium`, ...). Those imports belong behind the deferred imports in `get_client_class()`/`get_storage_class()`, which run only at first instantiation. This is what lets a plugin's dependencies stay uninstalled without breaking discovery for every other command.
    - **Base Contracts (`scrapers/base/`)**: Abstract base classes (`BaseScraperClient`, `BaseTrackedItem`, `ScrapeResult`, `BaseDataManager`, `BasePlugin`) defining the interfaces all plugins must implement, plus `ScraperSettings` (`scrapers/base/settings.py`) — the import-light dataclass modelling each scraper's config `settings` block (the shared base every plugin inherits, subclassable per store), together with the value normalizers and the `execution_interval` → systemd-`OnCalendar` mapping.
    - **Registry (`scrapers/registry.py`)**: A unified `ScraperRegistry` that replaces the old separate `ScraperFactory` and `DataManagerFactory`. Handles URL-to-plugin resolution, lazy client/manager instantiation, auto-discovery of plugins via `pkgutil`, and the import-light **settings resolvers** (`resolve_interval_status`, `resolve_timer_directives`, `resolve_log_retention`) that read a scraper's config `settings` block — falling back to the plugin defaults — without loading its storage stack, so the shell helpers and `--status` can use them cheaply.
  - **Orchestration (`orchestrator.py`)**: `ScrapingOrchestrator` runs a store-agnostic execution loop over targets and products, handling per-target file locks (`locks.py`), retries, and notification dispatch.
  - **Notifications (`notifier.py`)**: Apprise wrapper; used both for price-drop alerts and to notify the user of critical failures.
  - **Configuration & Validation (`constants.py`, `utils.py`)**: Centralized constants and environment/file health checks.
  - **Terminal UI (`tui.py`)**: Manages the interactive progress bar during sleep intervals using the Strategy pattern (`ExecutionStrategy`): `InteractiveExecutionStrategy` renders the Rich progress bar; `SilentExecutionStrategy` is used with `--quiet` (and by the systemd service).
  - **CLI Tools (`ping.py`, `status.py`)**: Dedicated scripts for specific CLI commands to maintain separation of concerns.
  User configuration is completely externalized to `.env` (`NOTIFICATION_URLS`, comma-separated Apprise URLs) and one `config/<plugin>.json` per scraper (the filename comes from the plugin descriptor). That file holds an optional top-level `settings` object — per-scraper preferences such as `execution_interval` and `log_retention_days` — alongside the list of tracked products (e.g. `config/skroutz.json`). The JSON config also doubles as state: the scraper writes the latest price and check timestamp (stored in UTC) back onto the *product rows* via `update_item`. The `settings` block, by contrast, is **read-only user input** — the application never writes it back (there is intentionally no settings write path / no stateful settings); machine-owned runtime state lives on item rows, never in `settings`. Settings are *read* at config/schedule time through the registry's `resolve_*` helpers and at scrape time through the data manager's `get_settings()`. The application is designed to run silently as a background task, leveraging a local Python virtual environment (`venv`) to isolate dependencies.

## Building and Running
The application is primarily intended for automated background execution but provides wrapper scripts for easy management and manual testing. There is no test suite or linter configured. All execution goes through the venv (`./venv/bin/python3`); the wrapper scripts handle this for you.

- **Installation:** Execute `./install.sh`. This script creates the Python virtual environment, installs the **core framework** dependencies from the root `requirements.txt`, then installs the **per-plugin** dependencies (each plugin's own `scrapers/<plugin>/requirements.txt`, if present) for every plugin being provisioned, and generates one systemd user timer+service pair **per registered plugin** (`<plugin>-scraper.{service,timer}`), each invoking `run.sh --quiet --<plugin>`. The root `requirements.txt` is framework-only (apprise, filelock, python-dotenv, rich, typing_extensions); transport/parsing libraries that only one scraper needs (e.g. `tls-client`, `selenium`) live in that plugin's own `requirements.txt`, so an install that skips a heavy scraper never pulls its dependencies. Each plugin's `[Timer]` *trigger* (cadence) is the user's `settings.execution_interval` when set and valid, otherwise the plugin's own default from `get_timer_directives()` — so a new store can declare its default schedule (and a user can override it per config) without editing any shell script. The config-aware `list_plugin_timer_directives` resolves the effective cadence, and `install.sh` and `scripts/schedule.sh` both render the unit files through the shared `write_plugin_units` helper in `common.sh`. `RandomizedDelaySec` (a hardcoded 180s/3-minute randomized startup delay) and `Persistent=true` are framework-managed in `install.sh` for every plugin and are intentionally *not* plugin-configurable (any such keys a plugin returns are dropped). Pass `--<plugin>` (e.g. `./install.sh --skroutz`) to (re)provision just one plugin's unit — installing only the core + that plugin's dependencies. Running a scraper whose dependencies were never installed fails that target cleanly with a `PluginDependencyError` pointing at `./install.sh --<plugin>`, rather than crashing the run.
- **Automated Execution:** Handled automatically by the per-plugin systemd timers (`<plugin>-scraper.timer`, e.g. `skroutz-scraper.timer`).
- **Service Management:** Use `./scripts/disable.sh` to temporarily stop and disable the background timer(s), `./scripts/enable.sh` to resume them, and `./scripts/stop.sh` to forcefully kill a running execution. With no flag these act on **all** plugins; pass `--<plugin>` (e.g. `./scripts/disable.sh --skroutz`) to target a single store. `./scripts/schedule.sh` re-applies each scraper's configured `execution_interval` to its installed timer (rewrites the unit, then `daemon-reload` + `try-restart`); run it after editing that setting — a missing config or an unsupported value is reported and that timer left unchanged. The teardown commands (`disable`/`stop`/`uninstall`) validate a targeted `--<plugin>` against the **union of registered plugins and installed units** (`is_known_target` in `common.sh`), so they can still tear down a *leftover unit of a plugin removed upstream* while rejecting a genuine typo; `install`/`enable`, by contrast, require the plugin to exist in the registry (they need code to run, not just a unit to remove). `./update.sh` (repo root) updates the installation; it derives which plugins to reprovision from the units already installed (globbed via `list_installed_plugins`), so it **preserves the user's original selection** — a single-plugin install stays single, and a store newly added in the pulled version is *not* auto-provisioned (and its dependencies not installed) until an explicit `./install.sh --<store>`. A plugin that was removed/renamed in the new version is skipped rather than aborting the update. With nothing installed, update falls back to a full install. All shell scripts are POSIX `sh` (verified against `dash`) and source shared helpers from `scripts/lib/common.sh` (colors, plugin enumeration via the registry, and systemd helpers), so adding a new store requires no changes to any existing script.
- **Manual Execution:** Execute `./scripts/run.sh` to run the scraper interactively and view output logs in the terminal. Flags: `--quiet` (no console output) and one dynamically generated flag per registered plugin (e.g. `--skroutz`) to run only that scraper. The `--ping` and `--status` flags must be used alone.
- **Direct Invocation:** `./venv/bin/python3 src/core/main.py` — note that `main.py` inserts `src/core/` into `sys.path`, so all intra-project imports are written relative to `src/core/` (e.g. `from scrapers.registry import ...`), not `src.core.scrapers...`.
- **Help Message:** Execute `./scripts/run.sh --help` to print the help message and view all available script arguments.
- **Testing Notifications:** Run `./scripts/run.sh --ping` to send a test payload and verify that the Apprise URLs in your `.env` file are configured correctly.
- **Health Checks:** Run `./scripts/run.sh --status` to perform a comprehensive health check. This validates the configuration (products and notification settings), checks for available script updates, and verifies the status of the background systemd service and timer. It footnotes per-scraper **settings issues** — a live timer whose schedule has drifted from the configured `execution_interval` (pointing at `./scripts/schedule.sh`) and an out-of-range `log_retention_days` — and surfaces **orphaned units** — installed `*-scraper.{timer,service}` units whose plugin is no longer registered (removed/renamed upstream) — each in its own red panel with the `./scripts/uninstall.sh --<plugin>` command to remove them, since they never run and are otherwise invisible to the per-plugin status loop.
- **Uninstallation:** Execute `./scripts/uninstall.sh` to cleanly stop and disable the systemd services and remove the virtual environment (user data is preserved). It removes every installed `*-scraper.{service,timer}` unit (robust to plugins deleted from the source tree). Pass `--<plugin>` (e.g. `./scripts/uninstall.sh --skroutz`) to drop only that plugin's units while keeping the venv and other plugins.
- **Exit Codes:** The script utilizes specific exit codes to indicate failure states when running as a service (these can be easily viewed using the `--status` flag):
  - **`15`**: Indicates an issue with a plugin's products config file, e.g. `config/skroutz.json` (file missing, wrong permissions, or invalid JSON).
  - **`16`**: Indicates an issue with the `.env` file configuration.
  - **`17`**: Indicates the scraper was blocked by the server due to rate limits.
  - **`42`**: Indicates that a specific scraper target did not start because another instance of it is already running (file lock timeout).
  - **`130`**: Indicates that the script was interrupted (e.g., via Ctrl+C or system termination).

## Development Conventions
- **Scraping Practices & Rate Limiting:** The scraper intentionally paces requests using a base delay (20s) plus randomized jitter (1-5s) between product checks to avoid triggering Skroutz's anti-bot protections. Concurrency is avoided to maintain a low profile. Do not "optimize" this away.
- **Data Integrity:** Updates to the `config/*.json` files (such as logging the latest price and check timestamp) must go through the existing atomic save mechanism to prevent file corruption in case of unexpected interruptions.
- **Error Handling & Logging:** When running silently in the background, exceptions are caught and logged to the relevant `errors.txt` — `logs/<target>/errors.txt` for a specific scraper (e.g. `logs/skroutz/errors.txt`), falling back to the root `logs/errors.txt` for top-level crashes with no target context. Background stdout is written to `logs/<target>/output.log`, which rotates daily at midnight UTC and keeps the scraper's configured `log_retention_days` of files (default 7; the surplus is pruned at the next rotation). Timestamps in these logs are recorded in UTC. If critical errors occur (like repeated scraping failures or a total crash), the script utilizes the Apprise notifier to alert the user of the failure.
- **Code Style:** Standard Python object-oriented practices are used with a focus on the Single Responsibility Principle, Dependency Injection, and the Open/Closed Principle. Logic is strictly separated into focused modules. Adding support for new stores requires creating a new plugin package under `scrapers/` implementing `BasePlugin`, `BaseScraperClient`, `BaseTrackedItem`, and `BaseDataManager` — no existing files (Python *or* shell) need to be modified. If the new store needs libraries beyond the core framework, list them in `scrapers/<plugin>/requirements.txt` (keeping their `import` statements behind the deferred `get_client_class()`/`get_storage_class()` calls per the import-light contract); to run on a non-default schedule, override `get_timer_directives()`. A new store inherits the shared `settings` (`execution_interval`, `log_retention_days`) for free; to add a store-specific setting, subclass `ScraperSettings`, return it from `get_settings_class()`, and add a `resolve_<setting>` wrapper over `_resolve_setting` in `scrapers/base/settings.py` (it owns the file read + unset/invalid/ok status machine). Store-specific settings must be **read-only user input** (no write-back) and are consumed at scrape time via the data manager's `get_settings()`. Configuration and state are decoupled from the main execution logic.
